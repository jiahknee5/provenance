Architecture & Data-Process Diagrams

The product is a system of record for claims. Five modules form a closed trust loop — generate → verify → optimize → send → monitor → re-verify — with the Assurance Lab proving the verifier works. Each module below shows its architecture (structure) and its data process (flow from system input to output).

◆

One closed loop: generate, verify, optimize, monitor — truth is the boundary

The Claims Library feeds generation and the Gate. The Gate clears only entailed, permissible claims; cleared variants are the only arms the Optimizer may pull. The Drift Monitor re-triggers the Gate when a source changes. The Assurance Lab continuously attacks the Gate so reliability is a measured number.

ArchitectureThe claim system of record

The Library feeds both generation and the Gate; every verdict lands in the ledger + audit trail; Drift and Assurance wrap the Gate.

Data processThe claim lifecycle

Falsehoods never enter the reward loop, so the Optimizer cannot converge to a higher-reward lie — constrained optimization as a structural fix for reward-hacking.

The single most original bet: bounding the Optimizer's action space to verified outputs makes reward-hacking structurally impossible — the policy can't win by lying because lies are never on the menu.
1

Claims Library — messy docs become a versioned, source-bound claim graph

Input · substrate

Atomic-claim extraction + evidence-binding + MLR approval turn approved documents into a queryable claim–evidence graph. Generation and reps pull only from the approved, in-date set — a retired claim is not selectable.

ArchitectureExtract → bind → approve → graph

Output consumers: Generation, The Gate (evidence retrieval), and Drift Monitor (source versions).

Data processSource doc → queryable claim

An unstructured document becomes versioned, source-bound, approved claim nodes; rejects are quarantined and not selectable.

2

The Gate — every claim is cited or killed before it can be sent

Verification · core

Decompose → bind → NLI + calibrated diverse-judge ensemble → compliance rules → cheap-first router-referee cascade. Replaces the gameable single LLM-judge with calibrated selective prediction. Output: a green / amber / red claim ledger.

ArchitectureCalibrated-ensemble verification

Cheap NLI runs first; only the uncertain claims escalate to the full ensemble — cost scales with difficulty.

Data processDraft → governed message + ledger

Green = cite · amber = repair / disclaimer · red = block. Legal sets the rules once; every message is checked in seconds.

3

The Optimizer — it optimizes conversion, but only inside the truth

Decisioning · most novel

A contextual bandit (Thompson / LinUCB) over a constrained action space of verified + cleared variants only. Hierarchical priors for cold-start; doubly-robust off-policy eval. The truth boundary is the guardrail on the RL loop.

ArchitectureBandit over verified arms only

An unverified / blocked variant never enters the action pool — the anti-Goodhart boundary.

Data processToward per-segment convergence

Each micro-segment auto-converges to its best verified message — continuous lift, zero added liability.

4

Drift Monitor — when a source changes, only the affected claims re-verify

Temporal

A claim-dependency graph + change-data-capture propagates invalidation to exactly the affected claims, re-runs only those through the Gate, and auto-pauses a now-false claim or unblocks a now-allowed one. Incremental cache-invalidation, for truth.

ArchitectureDependency graph + re-verification

Re-verification reuses the Gate on the affected subset only — not the whole corpus.

Data processFlow on a source change

Now-false → paused before it sends; now-permitted → unblocked everywhere instantly.

5

Assurance Lab — attack your own checker, then publish the number

Proof · eval

Synthetic adversarial traps (number-drift, unsupported superlative, false equivalence, true-but-unsayable) run through the Gate; decomposed reliability — catch-rate at fixed false-reject, paraphrase robustness, ECE, severity buckets — vs. a single-LLM-judge baseline. A wind tunnel for the checker.

ArchitectureAdversarial self-test harness

The harness exercises the real Gate and compares it against the baseline judge on the same traps.

Data processClaims → audited metric

Turns trust into a number a risk committee can act on — assurance itself shortens enterprise sales cycles.
