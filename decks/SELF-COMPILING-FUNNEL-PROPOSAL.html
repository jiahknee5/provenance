Gauntlet Capstone · Proposal · Showcase Jun 29

The Provable Funnel

AI outbound that can't say anything it can't prove — every claim decomposed, cited, verified, or killed before it sends.

Thesis. Generation is solved; verification is the bottleneck (Karpathy). Every AI marketing tool sprays unverifiable slop — and in regulated outbound (ag, food, health, finance) a single false claim is legal liability, not a typo. We build a generation engine with a hard verification gate: each message is split into atomic claims, every claim must bind to a real source and pass an entailment check, and anything unsupported — or true but not legally sayable — is repaired or blocked. Built on a live lead-gen spine, generalizing to any company.

The reframe (the spiky POV)

The model can't say what it can't prove. Cited, or killed.

Everyone is racing to make AI write more outbound. The frontier problem is the opposite: making it write only what's true, sourced, and legally sayable. LLMs embellish — that's exactly what makes marketing copy persuasive and dangerous. We invert the burden of proof: a sentence doesn't ship because it sounds good; it ships because every atomic claim in it passed verification against a real source. This is the trust/verification gap — applied to generation, in a domain where being wrong has a price.

The unit of progress: hallucination catch-rate at a fixed false-reject rate and cost — how many false/unsayable claims we stop before send, without nuking the good copy.

The problem — and why it's technically hard

"Only say what you can prove" sounds like a content rule. As an AI system it's a stack of hard, currently-open problems:

  • Claim decomposition from fluent text. Persuasive copy hides multiple claims in one sentence, mixes checkable assertions with puffery ("the best"), and implies things it doesn't state. Splitting generated prose into atomic, checkable claims — and knowing which are factual vs. opinion — is itself unsolved.
  • Verification when the judge is unreliable. The obvious verifier — "ask an LLM if the claim is supported" — fails >50% of bias tests and is gameable. You need a calibrated entailment check (NLI model + judge ensemble), not a single LLM yes/no, because false-confidence here ships a lie.
  • Repair without killing the copy. Stripping an unsupported claim usually breaks fluency and persuasion. Constrained regeneration that preserves the message while only asserting verified facts is a hard generation problem — and the adversary is the generator's own drive to embellish (the RLVR reward-hacking analogue).
  • True ≠ sayable. Some claims are factually true but legally blocked (the UTK feed-efficiency numbers, under hold). Claimability is orthogonal to truth — a second gate, encoded as a per-domain regulatory ontology, not a fact check.
  • Cheap enough to run on every sentence, per lead, at scale. You can't run a five-model ensemble on every claim of every message. You need a cheap first-pass that escalates only uncertain claims — a verification cascade with a cost budget.

Crucially, this is measurable: build messages with known-supported and known-false claims and you can score the verifier's precision/recall, catch-rate, and false-reject directly — the ground truth marketing usually lacks.

Project direction

Primary fit · Direction A

Combine ML with LLM applications. An LLM generates persuasive copy; a trained verification stack (NLI entailment model + a calibrated judge ensemble + a claimability classifier) decides, claim-by-claim, what is allowed to survive. The LLM does what ML can't (write); the ML stack does what the LLM can't (reliably know what it's allowed to assert). Remove the verifier and you have generic AI slop — the verifier is the product.

It also touches B and C — but lead with A

  • One problem, many ways (C): we benchmark verification approaches head-to-head (single-judge vs. NLI vs. ensemble vs. self-consistency) and report what actually catches hallucinations cheapest — a real comparative spine, but secondary to building the gate.
  • Agent design (B): the funnel still compiles per-lead and a learned policy still routes channel/timing — but those are the ambition thread, not the headline. The headline is provable generation.
3

AI is the primitive, not a bolt-on

The tell: nothing ships unverified. The generator is a candidate-writer; the verification gate is the authority. Delete it and the system is a worse mail-merge. The human sets the source set and the regulatory rules; the models do the rest.

lead facts ─► GENERATOR (LLM, per-lead) ─► draft message │ ┌──────────── VERIFICATION GATE — the primitive ────────────────┦ │ DECOMPOSE draft → atomic claims │ │ ▼ │ │ BIND each claim ↦ source row (citation ledger) │ │ ▼ │ │ VERIFY NLI entailment + judge ensemble (calibrated) │ │ ├─ supported + sayable ──► KEEP (green, cited) │ │ ├─ unsupported ──────► REPAIR / DROP (amber) │ │ └─ true but not sayable ► BLOCK (red, regulatory gate) │ └────────────────────────────────────────────┦ ▼ provable message ─► HITL approve ─► send cost control: cheap NLI first-pass → escalate only uncertain claims to the ensemble (cascade)
  • Decompose. An LLM + rules split the draft into atomic claims, tagging each factual / puffery / implied. Coverage is measured (no claim escapes the ledger).
  • Bind. Each factual claim is matched to a candidate source row from the lead's grounded facts (the existing citation-ledger idea, hardened).
  • Verify. A calibrated stack — NLI entailment model first, judge ensemble on the uncertain tail — decides supported / unsupported, with a reliability diagram so the gate's confidence is trustworthy (not a single LLM's say-so).
  • Gate & repair. Supported+sayable survives; unsupported is repaired or dropped via constrained regeneration; true-but-unsayable is blocked by the regulatory ontology. The output is a message where every surviving sentence is cited.

The loop that wraps it — constrained self-optimization

A contextual-bandit policy proposes message variants per segment (angle, which proven claim to lead with, channel, timing); every variant passes the same verification gate before it can be sent; outcomes (reply / convert, with unsubscribe as a hard negative) update the policy. The optimizer maximizes conversion only within the space of verified messages — so it structurally cannot reward-hack into exaggeration. The verifier is the guardrail on the RL loop; the loop is what makes verified copy win. This is the same auto-A/B technique Netflix/Meta use, plus the truth constraint they lack.

Scope — what "working" means in two weeks

Anchor on tenant #1 (lysozyme/LysoReach) — real leads, real sources, a real regulatory hold (UTK) — so verification has genuine, checkable ground truth.

Minimum shippable — must hit
  • End-to-end on tenant #1: generate a per-lead message → decompose → bind → verify → repair → the live claim ledger (green/amber/red).
  • Verifier = NLI entailment + a calibrated check; report catch-rate, false-reject, precision/recall on a built test set of true/false claims.
  • The regulatory gate: block the UTK numbers on demand (true-but-held).
  • The adversarial demo: try to make it lie / embellish → the gate catches it. (Killer moment.)
Ambitious — scored on ambition
  • Verifier bake-off — single-judge vs. NLI vs. ensemble vs. self-consistency on catch-rate per dollar; pick the cost-optimal cascade.
  • Any-company generality — point it at a second tenant's sources + rules; show the gate transfers (the GTM-engine ambition).
  • Constrained self-optimization — a contextual-bandit auto-A/B over variants (angle / claim / channel / timing) that only ever tests verified messages, bootstrapped on the persona simulator. Optimizes conversion; can't win by lying.
  • Adversarial robustness — measure whether a generator pushed to embellish can slip claims past the gate (the reward-hacking test).

Why it's compelling — the recruit hook

  • It's the trust gap, made tangible. "We built the AI that can't lie in your outbound — every claim cited or killed" is a one-line pitch judges and recruits remember, and it sits on the hottest theme in AI right now.
  • Real stakes, real ground truth. Regulated outbound (ag/food/health/finance) makes verification a requirement, not a demo — and gives the project measurable precision/recall instead of vibes.
  • Head start no one else has. A live data spine, real sources, and a real legal hold already exist — day-one momentum while the team builds the frontier verifier.
  • The claim ledger demos like a magic trick — watch claims light up green/amber/red, try to make it lie, watch it refuse.

Hard parts / risks — the bet and where it breaks

The bet: a calibrated verification gate can catch false / unsayable claims at high recall without destroying the copy — cheaply enough to run on every message — beating "trust the LLM."

RiskWhere it breaks · mitigation
Verifier reliability
(top risk)		A single LLM judge is gameable and fails bias tests. Mitigation: NLI entailment + diverse-family ensemble + calibration (ECE, reliability diagram); report catch-rate vs. false-reject, never a bare yes/no.
Claim decomposition errorsCompound claims, implicature, and puffery evade extraction — a missed claim is an unverified claim. Mitigation: measure decomposition coverage explicitly; start with factual-assertion extraction, handle puffery as a separate class.
Repair breaks the copyDropping a claim can wreck fluency/persuasion. Mitigation: constrained regeneration from the verified skeleton; show human-readable diffs; keep a fluency check in the loop.
Adversarial generatorEmbellishment is what makes copy good, so the generator pressures the gate (reward-hacking analogue). Mitigation: adversarial eval as a deliverable; report slip-through rate, don't claim it's zero.
Optimizer reward-hackingA self-optimizing loop maximizing replies normally drifts toward exaggeration (Goodhart). Mitigation: the verifier bounds the search space — only verified variants compete — plus brand/tone guards and unsubscribe as a hard negative reward.
Sparse reward / cold-startReplies are rare and slow, new segments have no data, and exploration burns real leads. Mitigation: bootstrap on the LLM-persona simulator; cheap proxies (opens) shape the slow signal; a hierarchical bandit shares across similar segments; learn off-policy from logs.
Cost at scaleEnsembling every claim is too expensive per lead. Mitigation: cheap NLI first-pass, escalate only uncertain claims (cascade) — and measure catch-rate per dollar.
Regulatory ontology is manual"True but unsayable" rules must be authored per domain. Mitigation: encode as data, human-confirmed; start with tenant #1's known holds (UTK), generalize later.
