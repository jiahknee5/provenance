Form → next visit · identity & personalization

Deck walkthrough (Deliverable 8): Use this slide after “form submit” in the attribution story. Answers: “How does the website know them next time?” — not from reading their phone; from identity we set at submit (cookie + token + email merge).

Walkthrough slide · identity

Form submit = stop being anonymous

Cold first visit: browser + IP + UTM only. After the form: we save a profile row and bind the browser (or link) to that person on every return visit.

How we recognize them on the next visit

MethodWhen it worksDemo / build
First-party cookie gauntlet_vidSame browser, cookies not clearedSet on form POST · read on every page load
Signed URL tokenEmail, QR, SMS — any device/w/{token} · Gauntlet day primary path
Email mergeDrew drip, night school row same emailUpsert into same recipient_id
Meta / Google IDsSupplement only; same browser + prior pixelAttribution debugger (simulated Week 1)
Login / OAuthExplicit consentStretch · Week 2+

Implementation spec (cohort web app)

// On form POST (server)
const visitorId = crypto.randomUUID();
await db.recipients.upsert({ recipient_id: visitorId, email, ...fields, utm, submitted_at });
setCookie('gauntlet_vid', visitorId, { maxAge: 365*24*3600, secure: true, sameSite: 'Lax' });
firePixelEvents({ meta: 'Lead', ga4: 'generate_lead' }); // simulated in demo
redirect(`/welcome?token=${sign(visitorId)}`); // optional: cookie + token once

// On every page load (server)
const id = cookies.get('gauntlet_vid') ?? verifyToken(searchParams.token);
const profile = await db.recipients.get(id); // RPG row
return renderPersonalizedPage(profile, track); // ungoverned | provenance

What we store at form submit (RPG row)

FieldPurpose
recipient_idStable internal key (cookie + token)
emailMerge Drew email · night school · future clicks
Form attributesRole, company, location, goals → personalization + Gate
utm_*, referrerAttribution debugger timeline
meta_fbp, ga_client_idOptional · simulated pixel correlation

Gauntlet day vs live form

Cohort already in Tom export
Form happened before demo · skip live form · pre-provision RPG rows · QR/magic link = same token mechanism
Live form on staging (optional)
Submit once in room → set cookie → reload → “it remembered me” · merge with batch personalization overnight

Honest limits (presenter crib)

ScenarioPersonalized return visit?
Same phone, same browser, cookie set at submitYes
New phone, no email/QR linkNo — anonymous again
Cleared cookies, no token in URLNo
Email or QR with signed tokenYes — best cross-device
“Read data from their phone” on first loadNo — not how the web works

Provenance angle: Identity resolution is separate from claim verification. Cookie/token tells us who — Gate tells us what we’re allowed to say to them.
